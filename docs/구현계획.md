# ClaFact 아키텍처 구현 계획

[설계 구조(architecture.md)](architecture.md)의 각 노드를 **어떤 기술로, 어떻게, 어디까지** 구현하는지에 대한 실전 적용 계획.

> **이 문서의 전제.** 이 아키텍처는 이미 **표준 라이브러리만으로 E2E가 도는 상태**다(테스트 85+ 통과). 그래서 "무엇을 새로 만드나"가 아니라 **"어느 스위치를 어떻게 켜나"**가 구현의 본질이다. 아래 모든 노드에 `[x]`(구현됨) / `[ ]`(할 일)를 표기한다.

## ✅ 착수 전 준비 체크리스트

> **먼저 읽기.** 구현을 시작하기 전에 무엇을 **받고·결정하고·익혀야** 하는지의 목록. 핵심은 아래 A~C가 준비되는 동안에도 **D(이미 준비된 것) 덕분에 개발이 멈추지 않는다**는 점 — 픽스처·Mock 으로 E2E 가 지금도 돈다.

### A. 받아야 할 것 (외부 — 없으면 그 부분이 막힘)

| 항목 | 어디서 | 담당 | 상태 | 없으면 막히는 것 |
|---|---|---|---|---|
| **HCX 키** | NCP CLOVA Studio | NLP | 다음 주 도착 | 탐지 2차 판별·정보 추출·설명 생성 |
| **KOSIS 운영계정 + 키** | KOSIS 활용신청 탭 | 검색·데이터 | **재발급 필요** | 실 조회·실 인덱싱 (개발계정 1,000회로는 부족) |
| **조선일보 데이터셋** | 클라비 제공 | PM | 대기 | 골든셋 대량 확장·EDA (단, **파일럿은 공개 기사로 지금 가능**) |

### B. 결정해야 할 것 (되돌리기 비싼 선택 — 착수 첫 주에)

| 결정 | 선택지 | 언제 | 기본값(권장) |
|---|---|---|---|
| **인덱싱 범위** | 도메인 제한 / 통합검색(경로 C) / 병행 | G1 (1주 말) | **병행** — 경로 C 주력 + 도메인 인덱스 백업 |
| **벡터DB 도입** | 인메모리(stdlib) / FAISS·Chroma | 필요해질 때만 | **인메모리** (수만 표 넘기 전엔 도입 안 함) |
| **임베딩 모델** | HCX v1 / v2 | 키 도착 후 | **실험(EXP-001)으로 결정** |
| **문장 분리** | 규칙 / kss | 데이터셋 도착 후 | **규칙** (정제율 측정 후 결정) |

### C. 익혀야 할 것 (담당별 사전 학습 — 키·데이터 기다리는 동안)

| 담당 | 익힐 것 | 왜 |
|---|---|---|
| **NLP** | CLOVA Studio 프롬프트 (JSON 강제 출력·`temperature=0`·폴백) | 키 오면 바로 붙이려면 프롬프트가 준비돼 있어야 |
| **검색·데이터** | 임베딩·cosine·리랭커 개념 + **검색어 생성** | 경로 C 성패가 질의 품질에 달림 (실측: "과수 농가 고령"만으론 원하는 표가 1위 아님) |
| **백엔드** | FastAPI 기초 + **한국 통계의 함정** (단위·시점·기준연도·잠정치) | 규칙이 곧 도메인 지식. API 는 순수 함수에 껍데기만 |
| **PM·평가** | 라벨링 프로세스(2인 교차·중재)·Kappa·평가 지표 | 골든셋은 사람만 만듦(오염 방지) |

### D. 이미 준비된 것 (위가 오면 꽂기만)

- **3개 스위치** — `KosisClient`·`LLMClient`·`Embedder` 프로토콜 전부 뚫림 (§3)
- **픽스처 E2E** — 키·데이터 없이 전 파이프라인이 지금도 돈다 → 개발 안 멈춤
- **자산 5종** (A1~A5) + **규칙 12개**(A2-0001~0012) + **하네스** + **호출 가드** 완성

## 0. 설계 철학 — 왜 이렇게 구현하나

| 원칙 | 의미 | 실전 효과 |
|---|---|---|
| **표준 라이브러리 우선** | 코어 파이프라인은 외부 의존성 0 (`requirements.txt`엔 streamlit 하나뿐) | 어디서나 `python`만 있으면 돈다. 배포·재현·인수인계가 쉽다 |
| **Protocol 스위치** | 외부 기술(KOSIS·HCX·임베딩)은 전부 인터페이스 뒤에 | 키·데이터 없이 개발하고, 도착하면 **한 줄 교체** |
| **Fixture ↔ Real 이중 구현** | 오프라인 픽스처와 실 API가 같은 인터페이스 | E2E가 항상 돈다. 시연 중 API 지연 위험 회피 |
| **판정은 결정적, LLM은 번역·설명** | 판정은 Python 코드(`verdict.py`), LLM은 탐지·추출·설명만 | 환각이 판정에 못 들어온다 = 우리의 핵심 주장 |

이 4개가 구현 방식을 전부 규정한다. **새 기술을 붙일 때도 이 원칙을 깨지 않는다** — 예: 벡터DB를 넣더라도 `Embedder` 프로토콜 뒤에 두어 스위치를 유지한다.

## 1. 기술 스택 전체 지도

| 레이어 | 지금 (구현됨) | 확장 (스위치로 교체) | 스위치 지점 |
|---|---|---|---|
| 문장 분리 | 정규식 (stdlib) | `kss`(한국어 문장 분리기) | `ingest.split_sentences` |
| 주장 탐지 | 규칙 필터 + 규칙카드 런타임 로드 | HCX 2차 판별 | `detect` + `detect_llm`(신규) |
| 정보 추출 | 정규식 파서 | HCX JSON 추출 | `parse` + LLM 슬롯 |
| 통계표 검색 | 별칭+키워드 / 문자 n-gram TF-IDF | HCX 임베딩 · **KOSIS 통합검색** · 리랭커 | `Embedder` 프로토콜 / 경로 C |
| 벡터 저장 | 인메모리 dict + cosine (stdlib) | FAISS or Chroma | `SemanticIndex` (교체) |
| KOSIS 조회 | Fixture(오프라인) / Http(실 API) | — (이미 실 API 구현) | `KosisClient` 프로토콜 |
| LLM | Mock (canned) | HCX (CLOVA Studio) | `LLMClient` 프로토콜 |
| 판정 | 결정적 Python | (LLM 안 씀 — 의도적) | `verdict.compare` |
| 설명 | 템플릿 (환각 0) | HCX Explainer + 환각 검사 | `run` 템플릿 → `explain`(신규) |
| 서빙 | Streamlit + stdlib http | FastAPI + Docker | `demo_server` → `api`(신규) |
| 자산 5종 | 전부 구현 (파일 기반 JSONL) | — | `assets/*`, `eval/harness` |

> **읽는 법:** 왼쪽 열(지금)은 **전부 돌아간다.** 오른쪽 열(확장)은 인터페이스가 이미 뚫려 있어, HCX 키·KOSIS 실데이터가 오면 **구현이 아니라 배선**이다.

## 2. 노드별 구현 명세 — 아키텍처 다이어그램 순서대로

각 노드: **파일 · 기술 · 입출력 계약 · 상태 · 구현 방식 · 함정.**

### 2.1 📰 Ingest — 전처리·문장 분리

| | |
|---|---|
| 파일 | `clafact/pipeline/ingest.py` |
| 기술 | `re`, `csv`, `json` (stdlib) |
| 입력 | JSONL/CSV 파일 경로 |
| 출력 | `list[Article]` (article_id·title·date·url·body·label·sentences) |
| 상태 | **[x] 완료** — 로더·정제·문장분리·중복URL제거·본문없음제외 |

**구현 방식:**
- **컬럼 별칭 흡수** (`FIELD_ALIASES`) — 실물 데이터셋 컬럼명이 달라도(`제목`/`기사 제목`/`title`) 코드 수정 없이 매핑. **데이터 도착 시 컬럼명만 추가.**
- **비본문 제거** — 기자명·저작권·이메일 정규식. 이메일을 먼저 지워야 "홍길동 기자 x@y.com" 줄이 기자명 패턴에 걸린다.
- **문장 분리** — 한국어 종결어미(`다./요./까?` 등)+구두점 기준. 소수점(`7.2%`) 보호.

**[ ] 확장:** 문장 분리를 `kss`로 교체하면 인용문·괄호 안 문장 처리가 좋아진다. **단, `kss`는 외부 의존성**이므로 코어 원칙(stdlib)을 깨려면 근거가 필요 → EXP로 규칙 분리 대비 오차를 측정한 뒤 결정. 교체는 `split_sentences` 한 함수만.

**함정:** 데이터셋 도착 전이므로 실 기사의 지저분함(표·이미지 캡션·관련기사 링크)은 아직 모른다. 도착 즉시 100건 샘플로 `clean_body` 정제율을 눈으로 확인할 것.

### 2.2 🔍 주장 탐지 — 규칙 필터 + LLM

| | |
|---|---|
| 파일 | `clafact/pipeline/detect.py` (규칙) + `detect_llm.py` (신규, LLM) |
| 기술 | 정규식 (stdlib) + HCX-DASH (경량 모델) |
| 입력 | 문장(str) |
| 출력 | `bool` (검증 가능 주장 후보 여부) + `which_rule()` |
| 상태 | **[x] 규칙 필터·규칙카드 런타임 로드** / **[ ] LLM 2차 판별** |

**구현 방식 (규칙, 완료):**
- 3중 필터: 숫자+단위 / 숫자+증감표현 / 숫자 없는 최상급(A2-0003).
- **규칙 카드가 곧 실행** — `rule_patterns()`가 `data/assets/rules/*.json`의 `pattern`을 런타임 로드. 카드 추가 = 코드 수정 없이 탐지 변경 (플라이휠의 핵심).
- 캐시 무효화는 **파일 목록 지문**(디렉터리 mtime 아님 — Windows에서 mtime이 안 바뀌는 버그를 이미 겪음).

**[ ] LLM 2차 판별 구현 계획 (`detect_llm.py`, 키 도착 후):**
```python
# 규칙 필터(재현율)가 통과시킨 후보 중 진짜 검증가능 주장만 남긴다(정밀도)
from clafact.llm import get_client
def is_verifiable_claim(sentence: str, client=None) -> bool:
    client = client or get_client()   # 키 있으면 HCX, 없으면 Mock
    out = client.complete(
        system="당신은 주장 판별기입니다. 검증 가능한 수치 주장이면 "
               '{"verifiable": true}, 의견·전망이면 false 를 JSON 으로만 답하세요.',
        user=sentence, model="HCX-DASH-002")   # 경량 모델(비용)
    return json.loads(out).get("verifiable", False)
```
- **파이프라인 배선:** `detect.is_candidate()`(규칙)로 1차 거른 뒤, `is_verifiable_claim()`로 2차. 규칙은 재현율(놓치지 않기), LLM은 정밀도(오탐 줄이기) — 역할 분리.
- **모델 선택:** 대량·경량이므로 `HCX-DASH-002` (문서 04 §6.2 모델 분업).

**함정:** LLM 2차 판별을 켜도 **규칙 필터를 앞에 유지**한다. 규칙이 후보를 좁혀야 LLM 호출 수(=비용)가 준다. 그리고 놓친 문장은 규칙 카드로 역산 → 플라이휠.

### 2.3 🧩 Claim Parser — 지표·시점·단위·모집단 추출

| | |
|---|---|
| 파일 | `clafact/pipeline/parse.py` (규칙) + LLM 슬롯 |
| 기술 | 정규식 (stdlib) + HCX-003+ (JSON 추출) |
| 입력 | 문장 + 기사 작성일 |
| 출력 | `ParsedClaim` (quantities·period·op·trend) |
| 상태 | **[x] 수치·시점·임계·추세** / **[ ] 지표·모집단 (LLM)** |

**구현 방식 (규칙, 완료):**
- **수치 추출** — 스케일(천/만/억/조) + 단위 정규식. 날짜 성분(`2024년`) 제외, 복합명사 가드(`1인 가구`의 '1인'은 수량 아님 — A2-0006).
- **시점 정규화 (A2-0005)** — `"올해"→작성연도`, `"지난해"→-1`, `"작년 3분기"→YYYY-Q3`. 우선순위: 절대 연월 > 절대 연분기 > 상대 연+분기 > … 반환형 `YYYY | YYYY-MM | YYYY-Qn`.
- **임계·추세** — `넘어섰다→gte`, `밑돌→lte`, `증가→up`. 활용형 주의(`섰≠서`).

**[ ] LLM 추출 구현 계획 (지표·모집단, 키 도착 후):**
```python
# 규칙으로 안 되는 것(지표명·모집단)만 LLM 에게. 규칙 결과는 교차검증 가드로.
def extract_slots(sentence: str, client=None) -> dict:
    out = client.complete(
        system='기사 문장에서 다음을 JSON 으로 추출: '
               '{"metric":"지표명","population":"모집단","comparison":"비교대상"}. '
               '없으면 빈 문자열.',
        user=sentence, model="HCX-005")   # 품질 중심 모델
    slots = json.loads(out)
    return slots
```
- **핵심 가드:** LLM이 뽑은 수치·시점은 **규칙 파서 결과와 대조**한다. 다르면 규칙을 신뢰(LLM 환각 방지). LLM은 규칙이 못 뽑는 **지표명·모집단만** 담당.
- `ParsedClaim`에 `metric`·`population` 필드 추가 → `run.py`가 매핑·판정에 사용.

**함정:** `temperature=0` 고정(멱등성). JSON 파싱 실패 대비 try/except → 실패 시 규칙 결과로 폴백하고 A4 기록.

### 2.4 🗺 KOSIS Retriever — 별칭 치환 → 통계표 검색 → 수치 조회

| | |
|---|---|
| 파일 | `retrieve.py`(경로 A) · `retrieve_semantic.py`(경로 B) · `kosis.py`(조회) |
| 기술 | 별칭사전 + 자카드 / 문자 n-gram TF-IDF + cosine / **KOSIS 통합검색** / HCX 임베딩 · 리랭커 |
| 입력 | 문장(질의) + 시점 |
| 출력 | `list[TableHit]` → `list[Evidence]` (조회된 수치) |
| 상태 | **[x] 경로 A + 조회** / **[ ] 경로 B/C/D 실험 · 실 인덱싱** |

**이 노드가 제품의 병목이다** — 매핑 실패 = 판단불가 = 제품 실패. 그래서 4파전 실험(EXP-001)으로 최선의 경로를 데이터로 고른다. **→ 처음부터 순서대로의 상세 구현은 [검색매핑_구현가이드.md](검색매핑_구현가이드.md).**

| 경로 | 방식 | 파일 | 상태 |
|---|---|---|---|
| **A** | 별칭 사전(A1) 치환 → 토큰 자카드 유사 | `retrieve.py::StatIndex` | [x] 완료 (기준선) |
| **B** | 문자 n-gram TF-IDF → cosine (→ HCX 임베딩으로 교체) | `retrieve_semantic.py::SemanticIndex` | [x] 프로토타입 / [ ] HCX 임베딩 |
| **C** | **KOSIS 통합검색 API** (검색어 → RANK) | 신규 `retrieve_kosis.py` | [ ] **G1 최우선** |
| **D** | C로 후보 생성 → 리랭커 재정렬 | C + HCX 리랭커 | [ ] 유력 |

**[ ] 경로 C 구현 계획 (인덱싱 0 — 문서 19 §4.6·§5.3):**
```python
# KOSIS 통합검색: 우리가 28만 표를 인덱싱하지 않고 KOSIS 검색엔진에 묻는다
# GET kosis.kr/openapi/statisticsSearch.do?searchNm=<검색어>&sort=RANK
def search_kosis(query_term: str, client) -> list[TableHit]:
    rows = client.integrated_search(searchNm=query_term, sort="RANK", resultCount=10)
    return [TableHit(r["TBL_ID"], r["ORG_ID"], r["TBL_NM"], r.get("STAT_NM",""),
                     score=None, prd_range=(r["STRT_PRD_DE"], r["END_PRD_DE"]))
            for r in rows]
```
- **핵심 이점:** 전수 인덱싱은 호출 예산상 불가능(28만 표 vs 1,000회 = 289배 초과, 문서 19 §5.3). 통합검색은 **주장당 1회**로 28만 표 전체가 사정권.
- **전제:** 자연어 문장이 아니라 **검색어**를 받는다 → Claim에서 좋은 검색어를 뽑는 **질의 생성**이 성패를 가름(이것이 EXP-001의 변수). 헤르메스/HCX가 `metric`+`population`으로 검색어 생성.
- **부가:** 응답의 `STRT_PRD_DE`/`END_PRD_DE`로 기사 시점을 포함 안 하는 표를 **조회 전에** 제거 → 예산 절감.

**[ ] 벡터DB 결정 (경로 B 확장 시):**
- **지금:** 인메모리 dict + cosine(stdlib). 통계표 수백~수천 개까지 충분히 빠르다.
- **언제 FAISS/Chroma:** 도메인 제한 인덱싱이 **수만 표**를 넘어 인메모리 검색이 느려질 때만. 그 전엔 도입하지 않는다(원칙: 필요해질 때까지 stdlib).
- **도입 시:** `SemanticIndex`를 통째로 교체하되 `search()` 시그니처 유지 → 파이프라인 무변경.

**[ ] HCX 임베딩 (`HcxEmbedder`):**
```python

        # CLOVA Studio 임베딩 v2 호출 → 밀집벡터. cosine 인터페이스 동일하게 맞춤
```

- **EXP-001의 목적:** 문자 n-gram은 글자 안 겹치는 동의어(`출생아` vs `새로 태어난 아기`)를 못 잡는다. 이 한계를 **측정해서** "왜 진짜 임베딩이 필요한가"를 데이터로 입증 → HCX 임베딩 채택 근거.

### 2.6 💬 Explainer — 근거·계산·한계 설명 + 환각 검사

| | |
|---|---|
| 파일 | `run.py` 템플릿(현재) → `explain.py`(신규, HCX) |
| 기술 | 템플릿 문자열(stdlib) → HCX-003/005/007 |
| 입력 | 확정된 판정 + Evidence(조회값) |
| 출력 | 설명 문자열 + 감사 추적 |
| 상태 | **[x] 템플릿(환각 0)·재현 패널** / **[ ] HCX Explainer·환각 자동 검사** |

**구현 방식 (완료):**
- **템플릿 설명** — 조회된 값만 문자열에 끼운다. 구조적으로 환각 불가능. `run.py`가 판정·근거·계산식·한계를 조립.
- **재현 패널(`audit.py`)** — 판정마다 코드버전·엔진·통계표·파라미터·**실 API URL(키 마스킹)**·적용 규칙 노출. 기업이 묻는 "감사 가능성"에 대한 답.

**[ ] HCX Explainer 구현 계획 (키 도착 후):**
```python
def explain(verdict, evidence, client) -> str:
    # 판정은 이미 확정됨. LLM 은 그걸 '자연스럽게 옮기기만' 한다.
    raw = client.complete(
        system="확정된 판정을 일반 독자용으로 설명하세요. "
               "제공된 근거 수치만 사용하고, '사실/거짓' 대신 '공식통계와 일치/불일치' 표현. "
               "새로운 수치를 만들지 마세요.",
        user=json.dumps({"verdict": verdict, "evidence": evidence}, ensure_ascii=False),
        model="HCX-005")
    # 환각 자동 검사: 설명 속 모든 숫자가 evidence 에 있는지 대조
    if hallucinated_numbers(raw, evidence):
        return template_explain(verdict, evidence)  # 환각 시 템플릿으로 폴백 + A4 기록
    return raw
```
- **환각 자동 검사가 핵심** — 설명 속 숫자를 정규식으로 뽑아 evidence의 값 집합과 대조. 근거에 없는 숫자가 있으면 그 건은 **실패로 계수**하고 템플릿으로 폴백.
- **표현 규칙** — "사실/거짓" 금지, "공식 통계와 일치/불일치"(문서 19 §7.2). KOSIS 값도 사후 정정될 수 있으므로.

| 컨테이너 | `Dockerfile` (stdlib http, `demo_server.py`) | uvicorn + FastAPI |

> **캐시·데이터 주의:** `data/cache/`(KOSIS 응답·예산 카운터)와 데이터셋은 `.gitignore`. 배포 시 캐시는 볼륨 또는 재생성.

## 6. 구현 순서 & 검증 — 무엇부터

의존성 순서. 각 단계는 **테스트 통과 + 하네스 지표 확인** 후 다음으로.

```
[지금 착수 — 키 불필요]
 1. A2-0012 잠정치 규칙   → verdict.py + 카드 + 테스트 (LST_CHN_DE 실 확인됨)
 2. 경로 C 검증(G1)       → retrieve_kosis.py + exp001 (브라우저 브릿지)
 3. 인덱싱 범위 결정      → 문서 19 §5.3

[HCX 키 도착 후]
 4. HcxClient 엔드포인트 검증 → 스모크 1건
 5. detect_llm 2차 판별   → 오탐률 측정
 6. parse LLM 추출        → 지표·모집단
 7. explain HCX + 환각검사 → 환각 0

[통합 — W5]
 8. FastAPI api.py        → /verify·/results·/review
 9. E2E 통합·시연 안정화

[상시 — 40일 내내]
 · 자산 축적 (별칭·규칙·골든셋) — 코드 아닌 운영 (문서 21·22)
 · 매 변경마다 하네스 → diff 확인 → 회귀 방지
```

**검증 원칙:** "하네스에 없는 지표는 없는 지표." 모든 구현은 하네스로 측정 가능해야 하고, 개선은 전 회차 대비 diff로 증명한다. 테스트 없는 규칙·모듈은 등록 불가.
