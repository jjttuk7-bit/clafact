# 소스 분류(Source Classification) 구현가이드

기획 원문: [24_소스분류_기획서](../../프로젝트_문서/24_소스분류_기획서.md) · 아키텍처: [architecture.md](../docs/architecture.md) · 선행 가이드: [검색매핑_구현가이드](검색매핑_구현가이드.md)

> **한 줄 목표.** 수치 Claim을 KOSIS로 보내기 **전에** `source_type` 8종으로 분류해, KOSIS가 아닌 Claim이 Verdict까지 흘러가 오판되는 것을 구조적으로 막는다. 최우선 지표는 **KOSIS_precision** — 놓친 건 나중에 규칙으로 줍지만, 잘못 보낸 건 오판이 된다.

---

## 0. 설계 결정 — 기존 원칙에 어떻게 맞추나

| 기존 원칙 | 소스 분류에서의 적용 |
|---|---|
| 표준 라이브러리만 | 분류기는 정규식+사전 (stdlib). LLM 2차는 `LLMClient` 프로토콜 뒤 (키 없으면 Mock) |
| 규칙 카드 = 실행 자산 | 라우팅 키워드 사전을 코드 상수가 아니라 **`data/assets/routing/*.json`으로 두고 런타임 로드** — 사전 추가 = 코드 수정 없이 동작 변경 (detect의 `rule_patterns()` 패턴 재사용) |
| Fixture ↔ Real 이중 구현 | 2단계 KOSIS 통합검색 확인은 기존 `search_kosis()`(경로 C) 재사용 — Fixture로 오프라인 개발, 클라우드에서 실검증 |
| 판정은 결정적 | 분류 결과가 애매하면 **UNKNOWN으로 회피**하고 사람 큐로. 억지로 KOSIS에 밀어넣지 않는다 (판단불가 철학의 확장) |
| 실패 1건 = 자산 1줄 | 오분류 보정은 A4 실패 레코드 → 라우팅 사전 보완 → 하네스 재평가 (플라이휠 그대로) |

**상태 머신 결정:** 새 State를 추가하지 않는다. 비-KOSIS 라벨은 기존 `판단불가` 계열로 종결하되 `ClaimResult`에 `source_type`·`route`·`route_reason`을 실어 UI가 "범위 밖(외부 근거원 후보)"과 "KOSIS 조회했으나 판단불가"를 구분 표기한다. 스키마 변경 최소화 — `schemas.py` 상태 머신은 단일 진실이므로 건드리지 않는다.

**파이프라인 삽입 위치:** `run.verify_sentence()` 안에서 `detect → parse` 직후, retrieval 앞.

```
detect.is_candidate() → parse.parse_claim()
→ source_classify.classify()            ← 신규
   ├─ KOSIS_DOMESTIC / KOSIS_BUT_COMPLEX → 기존 매핑·조회·판정 (변경 없음)
   ├─ OTHER_OFFICIAL / PRIVATE / PLATFORM / FORECAST → 판정 생략, 범위 밖 + 근거원 후보 명시
   ├─ NOISE → 결과에서 제외 (탐지 오탐으로 A4 기록)
   └─ UNKNOWN → 사람 검토 큐
```

**구현 규칙:**
- 로드는 detect와 같은 **파일 목록 지문(`_signature`) 캐시** 방식 — 디렉터리 mtime은 Windows에서 안 바뀐다(이미 겪은 버그). 공용 헬퍼로 뽑아 재사용 권장.
- `priority`가 곧 충돌 해소 순서다. "내년 성장률 3% 전망"은 `경제성장률`(OTHER_OFFICIAL)과 `전망`(FORECAST)이 동시에 걸린다 → **FORECAST가 이긴다.** 전망 판정은 키워드 + `parse.normalize_period()` 결과가 기사 작성연도보다 미래인지로 이중 확인.
- **알려진 함정(이미 겪음):** "가구"는 단위이자 주제어다. 키워드 매칭도 query_gen의 교훈대로 **숫자에 붙은 단위 스팬은 매칭 대상에서 제외**해야 "150만 가구"의 '가구'로 population_household를 오발동하지 않는다… 는 반대고, 오히려 발동해야 맞는 사례도 있다 → v0.1에서는 단순 포함 매칭으로 시작하고, 오분류 사례가 쌓이면 사전에 `exclude_patterns` 필드를 추가한다 (성급한 정교화 금지).

## Step 2. 분류기 본체 — `clafact/pipeline/source_classify.py` (신규)

```python
@dataclass
class RouteDecision:
    source_type: str          # 8종 라벨
    domain: str               # employment_labor 등 ("" 가능)
    route: str                # KOSIS_RETRIEVAL | NON_KOSIS_QUEUE | REVIEW_QUEUE | DROP
    confidence: float         # 0.0~1.0
    matched_keywords: list[str]
    expected_source: str      # "KOSIS 고용·노동 통계" / "한국부동산원" 등
    risk_type: list[str]      # KOSIS_BUT_COMPLEX일 때 함정 규칙 ID
    reason: str               # 사람이 읽는 한 줄

def classify(sentence: str, parsed: ParsedClaim, article_date: str,
             rules: RoutingRules | None = None) -> RouteDecision: ...
```

**분류 순서 (결정적, 위에서 아래로):**

1. **NOISE 가드** — `parsed.quantities`가 비었거나 전부 날짜·순번·나이 성분이면 NOISE. (detect·parse가 대부분 걸러주지만, 새어 들어온 것의 최종 방어선)
2. **FORECAST 가드** — forecast_markers 매칭 + 시점이 미래 → FORECAST_OR_OPINION.
3. **사전 매칭** — 전 도메인 키워드를 훑어 매칭된 (domain, source_type) 후보 수집.
4. **충돌 해소** — 후보가 여럿이면 `priority` 순서로 결정. KOSIS 계열과 비-KOSIS가 동시에 걸리면 **비-KOSIS가 이긴다** (KOSIS_precision 우선 원칙의 코드화).
5. **무매칭 → UNKNOWN**, confidence 0.0, REVIEW_QUEUE.

**confidence 규칙(v0.1은 단순하게):** 단독 도메인 매칭 0.8 / 충돌 해소 경유 0.6 / FORECAST·NOISE 0.9 / UNKNOWN 0.0. 임계값 0.5 미만이면 route를 REVIEW_QUEUE로 강등. (수치는 G-SOURCE-1 실측 후 조정)

## Step 3. KOSIS 통합검색 확인 신호 — 기존 경로 C 재사용

기획서 2단계. **새로 만들 것 없음** — `retrieve_kosis.search_kosis()` + `query_gen.make_query()`를 그대로 쓴다.

```python
def confirm_with_kosis(decision: RouteDecision, sentence: str, client,
                       aliases=None) -> RouteDecision:
    # KOSIS 계열·UNKNOWN일 때만 호출 (비-KOSIS 확정 건은 호출 낭비)
    hits = search_kosis(make_query(sentence, aliases), client, top_k=10)
    ...  # 히트 유무·상위 tblNm 관련도로 confidence 가감, UNKNOWN→KOSIS 승격 가능
```

**호출 예산 주의 (중요):**
- 통합검색도 예산을 쓴다. **분류 확인용 호출은 KOSIS 계열 후보와 UNKNOWN에만** 제한하고, 어차피 다음 단계(retrieval)에서 같은 검색을 하므로 **결과를 넘겨 재사용**한다(같은 질의 2회 금지 — `CachedKosisClient`가 받쳐주지만 설계로도 막을 것).
- 개발망은 정부망 443 차단 → 로컬은 `FixtureKosisClient`, 실측은 클라우드(Streamlit Cloud). 기존 제약 그대로.
- v0.1 게이트(G-SOURCE-1)는 **확인 신호 없이 규칙만으로** 먼저 돌린다. 확인 신호는 규칙 분류의 오류율을 실측한 뒤 "얼마나 개선하나"를 EXP로 붙인다(애블레이션 가능하게).

## Step 4. 파이프라인 배선 — `run.py`

`verify_sentence()`에서 parse 직후:

```python
decision = source_classify.classify(sentence, parsed, article_date)
if decision.route == "DROP":            # NOISE
    return None                          # + A4 실패 레코드(탐지 오탐 후보)
if decision.route != "KOSIS_RETRIEVAL": # 비-KOSIS·UNKNOWN
    return ClaimResult(..., verdict=판단불가,
        source_type=decision.source_type,
        route_reason=f"범위 밖: {decision.expected_source} 필요")
# KOSIS 계열만 기존 매핑·조회·판정으로 진행 (기존 코드 무변경)
```

- `ClaimResult`에 `source_type`·`route_reason` 필드 추가. `audit.py` 감사 추적에도 `source_type` 한 줄 추가 (재현 패널에서 "왜 KOSIS로 안 갔나"가 보여야 함).
- **데모 효과:** 시연에서 "코스피 3000 돌파"를 넣으면 지금은 매핑 실패로 어색하게 판단불가가 뜨지만, 배선 후엔 **"거래소·시장자료 필요 — KOSIS 범위 밖"**이라고 정직하게 말한다. 시연 3막(정직한 회피)과 같은 서사.
- `KOSIS_BUT_COMPLEX`는 판정까지 가되 `risk_type`의 규칙 ID(A2-0007·0012·0013)를 결과에 표시 — 이미 구현된 가드들이 실제로 그 함정을 처리한다.

## Step 6. 평가 — 하네스 확장과 게이트

**파일:** `eval/harness.py`에 `eval_source_routing(rows)` 추가, `tests/test_source_classify.py` 신규.

| 지표 | 정의 | 목표(v0.1) |
|---|---|---|
| **KOSIS_precision** ★ | KOSIS로 라우팅한 것 중 진짜 KOSIS 비율 | **≥ 0.90** — 이게 깨지면 오판이 생긴다 |
| KOSIS_recall | 진짜 KOSIS 중 잡아낸 비율 | ≥ 0.70 (낮으면 사전 보강으로 회복 가능) |
| source_type_accuracy | 8종 전체 정확도 | 측정만 (기준선 확보) |
| UNKNOWN_rate | UNKNOWN 비율 | ≤ 0.20 (너무 높으면 사전이 빈약, 0이면 과신) |

**테스트 최소 세트:** 라벨 8종 각 1건 이상 + 충돌 케이스("내년 성장률 3% 전망" → FORECAST가 OTHER_OFFICIAL을 이김) + 사전 런타임 로드(파일 추가 → 재로드 → 동작 변경) + NOISE 가드. "테스트 없는 규칙은 등록 불가" 원칙 동일.

**게이트 G-SOURCE-1 완료 기준** (기획서 §9 그대로):
1. 경제면 Claim 50건 source_type 분포 생성
2. KOSIS / 비-KOSIS 후보 분리
3. UNKNOWN 사람 검토 큐 생성
4. KOSIS 오라우팅 사례 목록 (→ A4 레코드)
5. `routing_v01.json`을 근거와 함께 v0.2로 갱신

## Step 7. HCX 2차 분류 (키 도착 후 — 지금은 착수 불가)

detect의 "규칙=재현율, LLM=정밀도" 분업을 그대로 복제한다.

```python
def classify_llm(sentence: str, client=None) -> dict:
    client = client or get_client()   # 키 없으면 Mock
    out = client.complete(
        system='수치 주장의 근거 자료원을 분류하라. JSON만: '
               '{"source_type": "...8종 중 1...", "reason": "..."}',
        user=sentence, model="HCX-DASH-002", temperature=0)
    ...  # JSON 파싱 실패 → UNKNOWN 폴백 + A4 기록
```

- **호출 대상은 규칙이 UNKNOWN·저신뢰로 판정한 건만** — 비용 통제 + 규칙 우선 원칙.
- LLM이 KOSIS 계열로 승격시키려면 **통합검색 확인 신호(Step 3)와 합의해야** 승격. LLM 단독으로 KOSIS 라우팅 불가 (KOSIS_precision 방어선).

## 구현 순서 요약 — 무엇부터, 지금 가능한가

```
[지금 착수 — 키·데이터 불필요]
 1. routing_v01.json + source_classify.py + 테스트   ← 순수 stdlib, 오늘 가능
 2. run.py 배선 + ClaimResult/audit 필드              ← 기존 데모에서 즉시 보임
 3. 시드 골든셋 50건 (공개 기사로) + eval_source_routing
 4. classify_news_sources.py / review_source_routes.py
 → G-SOURCE-1 통과 = source routing rule v0.2

[클라우드/브라우저 브릿지]
 5. Step 3 확인 신호 실측 + 규칙-only 대비 애블레이션 (EXP-002로 명명 권장)

[HCX 키 도착 후]
 6. classify_llm 2차 분류 (UNKNOWN 축소)

[데이터셋 도착 후]
 7. 경제면 전수 분류 → 분포 리포트 → 문서 23 우선순위표 실측 검증
```

> **한 줄 요약.** 새로 만드는 건 분류기 한 파일 + 사전 한 파일 + 스크립트 둘뿐이고, 나머지(경로 C 검색·질의 생성·캐시·예산 가드·플라이휠·하네스)는 전부 이미 있는 부품의 재배선이다. 핵심 설계 판단은 세 가지 — **① 사전을 코드가 아닌 자산으로, ② 충돌 시 비-KOSIS 우승(KOSIS_precision 우선), ③ 애매하면 UNKNOWN 회피.**
