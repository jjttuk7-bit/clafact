# 검색·매핑 구현 가이드 — 처음부터 순서대로

**주장(Claim) → KOSIS 통계표 매핑을 어떻게 구현하나.** 제품의 병목이자, HCX 키 없이 지금 착수 가능한 영역.

> 관련: [architecture.md](../docs/architecture.md) · [구현계획.md](구현계획.md) §2.4 · 문서 19 KOSIS 백서 · 문서 12 확률적 계획
> 코드: `retrieve.py`(경로 A) · `retrieve_semantic.py`(경로 B) · `kosis.py`(조회) · `exp001_ab.py`(평가)

---

## 왜 이 문서인가 — 매핑이 제품의 병목이다

ClaFact 파이프라인에서 **매핑 실패 = 판단불가 = 제품 실패**다. 탐지·판정·설명이 아무리 좋아도, 주장에 맞는 통계표를 못 찾으면 아무것도 판정 못 한다. 그래서 이 영역에 가장 많은 실험과 결정이 몰려 있다.

**문제의 크기:** 주장 하나를 **KOSIS 28만 개 통계표 중 정확히 하나**에 연결해야 한다. 그런데 기사어와 통계어가 다르다 — 기사는 "과일 재배면적"이라 쓰고 KOSIS는 "농림어업조사 > 영농형태별 경영주 연령별 농가"라 부른다. 이 간극을 잇는 게 매핑이다.

**오늘의 실측 (2026-07-15, 통합검색 실호출):** "과수 농가 고령"으로 검색하니 1위가 우리가 원하는 표가 아니라 `농가수(시도/시/군/구)`였다. **→ 좋은 검색어를 뽑는 것(질의 생성)이 성패를 가른다**는 걸 데이터로 확인. 이 문서의 Step 1이 그래서 중요하다.

---

## 0. 전체 흐름 — 매핑이 하는 일

```
주장 문장                     "2024년 과수 농가 65세 이상 비율은 64.2%"
   │
   ▼ ① 질의 생성 (Step 1)      "농림어업조사 경영주 연령 과수"  ← 검색어
   │
   ▼ ② 후보 검색 (Step 2~4)    28만 표 → 상위 K개 후보
   │     경로 A 별칭+키워드 / B 임베딩 / C 통합검색 / D 하이브리드
   │
   ▼ ③ 재정렬 (Step 4)         리랭커로 후보 K개 → Top-3
   │
   ▼ ④ 통계표 확정             DT_1EA1019 (orgId=101)
   │
   ▼ ⑤ 수치 조회 (Step 7)      objL·ITM·시점 지정 → 근거 행들
   │
   ▼ ⑥ 근거 선택              분류/항목 매칭 → Evidence 리스트 → 판정으로
```

각 단계는 **독립적으로 교체 가능**하다 (경로 A→B→C→D는 스위치). 아래는 이 흐름을 **처음부터 순서대로** 구현하는 법.

## Step 0. 자료 구조와 인터페이스 (이미 있음 — 먼저 이해)

구현 전에 세 가지 계약을 안다.

**① 통계표 메타 (`tables_meta.json`):** 검색 대상. 각 표는
```json
{"TBL_ID":"DT_1EA1019", "ORG_ID":"101", "TBL_NM":"경영주 연령별 농가",
 "SURVEY":"농림어업조사", "KEYWORDS":"고령화 과수 영농형태 연령"}
```

**② `TableHit` (검색 결과):** `tbl_id · org_id · tbl_name · survey · score`. 모든 경로가 이 형태로 반환 → 경로를 바꿔도 하류 코드 무변경.

**③ `Evidence` (조회된 근거):** `tbl_id · value · unit · period · query_params · last_change_date`. 판정 엔진의 입력.

> **핵심 원칙:** 검색 경로가 무엇이든 출력은 `list[TableHit]`, 조회 출력은 `list[Evidence]`. 이 두 계약을 지키면 경로 A/B/C/D를 자유롭게 갈아끼운다.

## Step 1. 질의 생성 — 주장에서 검색어를 뽑는다 【신규, 최우선】

**왜 먼저인가:** 오늘 실측이 증명했다. 검색 엔진(경로 C든 B든)이 아무리 좋아도 **검색어가 나쁘면 못 찾는다.** "과수 농가 고령"은 나쁜 검색어였다.

**무슨 기술:** 규칙(지금) → HCX(키 후). Claim의 구조화된 필드에서 검색어를 조합.

**구현 방식:**
```python
# clafact/pipeline/query_gen.py (신규)
def make_query(claim) -> list[str]:
    """주장에서 검색어 후보를 만든다. 여러 개 → 각각 검색 후 융합(RRF).
    규칙 버전: 지표·모집단·조사명 힌트를 조합. LLM 버전: HCX 가 핵심어 추출."""
    parts = []
    if claim.metric:      parts.append(claim.metric)       # "고령화 비율"
    if claim.population:  parts.append(claim.population)    # "과수 농가"
    # 별칭 사전(A1)으로 기사어 → 통계어 확장
    expanded = alias.expand_terms(parts)                   # "농림어업조사 영농형태 연령"
    return dedupe([" ".join(parts), " ".join(expanded)])
```

**✅ 구현 완료 (`query_gen.py`, 2026-07-15).** `make_query(sentence, aliases)` — ① 별칭 치환 ② 숫자·날짜 스팬 제거 ③ 조사 벗기기·서술어 컷. 테스트 9건.

**측정 결과 (EXP-001 애블레이션, 문장째 vs 생성 검색어):**
| 경로 | Hit@3 문장째 → 생성 |
|---|---|
| A 키워드만 | 0.70 → **0.90** ▲ |
| B 의미(n-gram) | 0.80 → **1.00** ▲ |
| A+별칭 | 0.90 = (이미 치환 중) |
| 하이브리드 | 1.00 = |

노이즈를 걷어내니 약한 경로가 오르고 강한 경로는 손해 없다. **Step 1 가설이 데이터로 확인됨.**

**구현 중 잡은 버그:** "가구"는 단위이자 주제어다. 단위어를 단독 토큰으로 지우면 "서울의 1인 가구..." 검색어가 "서울"로 뭉개진다 → **숫자에 붙은 단위만 스팬으로 제거**(`RE_QTY_SPAN`)해 주제어를 살렸다.

**함정:** 검색어를 너무 길게(문장째) 넣으면 통합검색 RANK가 흐려진다. 짧고 핵심적인 통계어 위주로. **별칭 사전(A1)이 여기서 결정적** — "일자리를 구하지 못한"→"실업률 경제활동인구" 치환이 검색 품질을 좌우 (실측: 이 치환으로 함정 케이스가 잡힘).

**정직한 한계:** 현재 평가셋은 10건·표 5개라 포화(하이브리드 1.00) — **질의 생성의 진짜 효과는 경로 C(28만 표)에서** 드러난다. 지금 측정은 "해가 없고 약한 경로엔 이득"까지만 증명.

## Step 2. 경로 A — 별칭 + 키워드 검색 (기준선, 이미 완성)

**무슨 기술:** 별칭 사전 치환 + 토큰 자카드 유사도. 외부 의존성 0.

**구현 (`retrieve.py::StatIndex`, 완성):**
- 통계표명·조사명·키워드를 토큰화(단어 + 한글 2-gram) → 인덱스.
- 질의를 **별칭 사전으로 치환**(`aliases.substitute`) 후 토큰 겹침 점수 계산.
- 점수 = 교집합 / √(합집합) — 짧은 질의 편향 완화.

**역할:** **모든 실험의 기준선(baseline).** 경로 B/C/D는 이걸 이겨야만 채택된다 (문서 12 기술 계단 원칙). 지금도 픽스처에서 돈다.

**한계:** 글자가 겹쳐야 매칭된다. "출생아"와 "새로 태어난 아기"는 못 잇는다 → 이게 경로 B의 존재 이유.

## Step 3. 경로 B — 의미 임베딩 검색 (프로토타입 → HCX 교체)

**무슨 기술:** 임베딩(텍스트→벡터) + cosine 유사도. 지금은 문자 n-gram TF-IDF 프록시, 나중에 HCX 임베딩.

**구현 (`retrieve_semantic.py`, 프로토타입 완성):**
```python
class Embedder(Protocol):          # 임베딩 백엔드 스위치
    def embed(self, texts) -> list[dict[str, float]]: ...

class CharNgramEmbedder: ...       # 지금 (stdlib, 프록시)
class HcxEmbedder: ...             # 나중 (진짜 의미 임베딩) — 한 줄 교체

idx = SemanticIndex(tables, embedder=CharNgramEmbedder())   # → HcxEmbedder()
```

**구현 순서 (HCX 임베딩 도착 후):**
1. `HcxEmbedder.embed()` 구현 — CLOVA 임베딩 API 호출 → 밀집 벡터. `cosine` 인터페이스는 동일하게 맞춤.
2. `SemanticIndex(tables, embedder=HcxEmbedder())` 로 교체 — **하류 코드 무변경**.
3. EXP-001 재실행 → n-gram 프록시 대비 함정 케이스 개선 확인.

**핵심 — 왜 프록시로 먼저 하나:** 문자 n-gram은 동의어를 못 잡는다. 이 한계를 **측정해서** "왜 진짜 임베딩이 필요한가"를 데이터로 입증하는 게 EXP-001의 목적. 근거 없이 "임베딩이 좋겠지"로 도입하지 않는다.

## Step 4. 경로 C·D — KOSIS 통합검색 + 리랭커 【신규, 유력】

### 경로 C — KOSIS 통합검색 API

**무슨 기술:** 우리가 28만 표를 인덱싱하지 않고 **KOSIS 검색엔진에 검색어를 던진다.**

**엔드포인트 (2026-07-15 확인):**
```
GET https://kosis.kr/openapi/statisticsSearch.do?method=getList
    &apiKey=...&format=json&searchNm=<검색어>&sort=RANK&resultCount=10
```
응답 필드: `TBL_ID·TBL_NM·ORG_ID·STAT_NM·STRT_PRD_DE·END_PRD_DE·REC_TBL_SE(추천표)·STAT_DB_CNT(총건수)`

**✅ 구현 완료 (`retrieve_kosis.py` + `kosis.py::integrated_search`, 2026-07-15).** 테스트 9건.
```python
search_kosis(query, client, period="", top_k=10) -> list[TableHit]
```
- `KosisClient` 프로토콜에 `integrated_search()` 추가 — Fixture(오프라인 스탠드인) / Http(실 API) 스위치.
- `build_search_url()` — `statisticsSearch.do` URL, **키 마스킹** 공유.
- **시점 필터(`_covers_period`)** — 수록기간이 기사 시점을 안 맞으면 조회 전 제거(예산 절감), 모르면 유지(보수적).
- **느슨한 JSON 대비(`parse_json_tolerant`)** — 표준 파싱 실패 시 키 따옴표 보정 1회 시도(오늘 관찰한 형태 방어, 클라우드서 실포맷 확인 예정).
- 예산 로직을 `_call()`로 공통화 — 통합검색도 fetch 와 같은 예산·레이트 가드를 탄다.

**오프라인 검증됨:** `make_query`로 생성한 검색어 → `search_kosis` → 농림어업조사 표(DT_1EA1019)가 후보에 옴. **클라우드에서 `HttpKosisClient`로 스위치하면 같은 코드가 진짜 28만 표를 검색.**

**왜 유력한가:** 전수 인덱싱은 호출 예산상 불가능(28만 표 vs 1,000회 = **289배 초과**, 문서 19 §5.3). 통합검색은 **주장당 1회**로 28만 표 전체가 사정권. 인덱스 구축 비용 0.

**⚠️ 실측으로 확인한 함정 3가지 (2026-07-15):**
| 함정 | 내용 | 대응 |
|---|---|---|
| 검색어 품질 | "과수 농가 고령" → 원하는 표가 1위 아님 | Step 1 질의 생성이 필수 |
| 느슨한 JSON | 응답이 `{ORG_ID:"101"...}` (키 따옴표 없음) — 표준 `JSON.parse` 실패 | 관대한 파서 or 정규식 추출 필요 |
| 네트워크 | 개발망서 실호출 차단 | **클라우드에서** 호출 (§실행 환경) |

**부가 이점:** 응답의 `STRT_PRD_DE`/`END_PRD_DE`로 기사 시점을 포함 안 하는 표를 **조회 전에** 제거 → 예산 절감.

### 경로 D — C로 후보 생성 → 리랭커 재정렬

**무슨 기술:** 통합검색(C)이 뽑은 상위 후보를 **리랭커**(HCX)로 다시 정렬. 인덱싱 없이 리랭커 이점만.

**구현:** `search_kosis()` 결과 상위 10개를 리랭커에 `(주장, 통계표명)` 쌍으로 넣어 재정렬 → Top-3. **6주 일정에서 가장 비용 대비 효과가 클 후보** (자체 인덱스 0 + 리랭킹 품질).

## Step 5. EXP-001 — 네 경로를 데이터로 비교 【평가, 지금 가능】

**무슨 기술:** 매핑 평가셋(`mapping_eval.jsonl`)의 `gold_tbl_id` 기준 Hit@1 / Hit@3 / MRR.

**구현 (`exp001_ab.py`, 하네스 완성):**
- 현재 4경로 비교: A 키워드 / A+별칭 / B 의미(n-gram) / 하이브리드(RRF 융합).
- **경로 C·D 추가** → 6경로 비교로 확장.
- 케이스별 승패 표(`① Hit@1 / ③ Hit@3 / ✗`) — **어떤 유형에서 이기고 지는가**가 진짜 산출물.

**RRF (Reciprocal Rank Fusion):** 여러 경로의 랭킹을 순위 기반으로 융합. `score = Σ 1/(60+rank)`. 하이브리드의 근거.

**완료 기준 (게이트 G3):** Hit@3 ≥ 0.70. 미달이면 **도메인 1개로 축소**(문서 22 §3).

**지금 할 수 있는 것:** 프록시 임베딩으로 EXP-001을 지금 돌려 A vs B vs 하이브리드의 기준선을 잡아둔다. HCX 임베딩·경로 C가 오면 같은 하네스에 **스위치로 갈아끼워** 재실험.

---

## Step 6. 인덱싱 범위 결정 【의사결정, 첫 주】

경로가 정해지면 **얼마나 인덱싱하나**를 결정한다.

| 선택 | 방식 | 커버리지 | 비용 |
|---|---|---|---|
| **도메인 제한** | 고용·물가·농업만 `MT_ZTITLE` 하위 수집 | 제한적 | 수백~수천 회 (감당) |
| **경로 C** | 인덱싱 0 — 주장마다 통합검색 | 28만 표 전체 | 주장당 1회 |
| **병행 (권장)** | C 주력 + 도메인 인덱스 백업 | 전체 + 안전망 | 둘 다 |

**권장: 병행.** 경로 C가 실패해도 도메인 인덱스가 남고, 어차피 EXP-001 비교에 둘 다 필요.

## Step 7. 수치 조회 & 근거 선택 (이미 완성)

통계표가 확정되면 실제 수치를 꺼낸다.

**무슨 기술:** KOSIS 통계자료 API (`kosis.py::fetch_data`) + 분류/항목 필터.

**구현 (`retrieve.py::fetch_evidence`, 완성):**
- `client.fetch_data(org_id, tbl_id, prd_de=period)` → 행들.
- 분류(C1/C2)·항목(ITM) **부분 일치**로 필터 → `Evidence` 리스트.
- **축 배치 함정 (A2-0008):** KOSIS 결합 차원은 `"영농형태 : 과수"` 접두 형식. 연령은 분류(objL)가 아니라 **항목(ITM)** 차원 — 표마다 다르므로 분류/항목 메타를 먼저 조회해 축을 파악.
- `LST_CHN_DE`(최종수정일)를 함께 담음 → 잠정치 규칙(A2-0012).

**근거 선택 (`run.py::_pick_c1`):** 문장에 등장하는 분류값 우선, 없으면 "전국"(총계). 파생계산(비율·증감률)은 여러 행을 조합.

## 별칭 사전 (A1) — 검색을 돕는 핵심 자산

매핑의 정확도는 코드가 아니라 **별칭 사전**이 좌우한다. 검색 전 치환으로 기사어를 통계어로 바꾼다.

**스키마:** `alias(기사어) · canonical(통계어) · tbl_id · 항목경로 · 유래(실패ID) · 검증횟수`

**축적 루틴:** 매핑 실패(판단불가) 시 사람이 정답 표를 찾으면 그 자리에서 엔트리 등록. **3주차가 별칭의 황금기** — 매핑 실패가 대량 유입되는 주(문서 11 §4). 실패 1건 = 별칭 1줄.

**현재 5건 → 40일 300건.** 이게 후발주자가 못 베끼는 해자(문서 08).

```
 5. 실 인덱싱          도메인 제한분 수집

[HCX 임베딩·리랭커 후]

 8. 경로 D             리랭커 재정렬

[상시]

```

## 리스크 & 의사결정 요약

| 리스크 | 신호 | 대응 |
|---|---|---|
| 매핑이 안 됨 (제품 실패) | G3서 Hit@3 < 0.3 | 도메인 1개(농업)로 축소, E2E 우선 |
| 검색어가 나빠 못 찾음 | 오늘 실측된 문제 | Step 1 질의 생성 강화, 별칭 확충 |
| 통합검색 JSON 파싱 실패 | 느슨한 JSON | 관대한 파서 |
| 호출 예산 소진 | 개발계정 1,000회 | 운영계정 + 캐시 + 경로 C(인덱싱 0) |
| 실호출 차단 | 개발망 정부망 차단 | 클라우드 실행 |

> **한 줄 요약:** 인터페이스(`TableHit`·`Embedder` 스위치)와 기준선(경로 A)·평가 하네스(EXP-001)는 이미 있다. 남은 건 **질의 생성 · 경로 C 클라이언트 · HCX 임베딩 교체 · 리랭커**, 그리고 40일간의 **별칭 사전 축적**이다. 매핑은 코드보다 **자산**이 좌우한다.
