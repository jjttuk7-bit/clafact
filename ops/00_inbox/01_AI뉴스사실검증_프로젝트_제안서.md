## 제안 요약

> **잘 요약하는 AI가 아니라, 공식 통계라는 근거 안에서만 판정하는 AI를 만듭니다.**

본 프로젝트는 뉴스 기사에서 수치 기반 주장을 자동으로 탐지하고, KOSIS(국가데이터처) Open API의 공식 통계와 비교하여 사실 여부를 판정하며, 그 판정의 근거·계산 과정·한계까지 설명하는 End-to-End 사실검증 시스템의 PoC를 구축합니다.

단순 기사 검색이나 요약 수준을 넘어, **검증 가능한 주장 탐지 → 공식 통계 조회 → 비교 검증 → 설명 생성 → 검증자 리뷰**로 이어지는 완결된 업무 흐름을 목표로 합니다.

| 클라비가 요청한 것 | 본 제안의 접근 | 산출물 |
|---|---|---|
| 검증 가능한 주장 탐지 | 규칙 필터 + LLM의 2단계 탐지로 수치·비교 표현 중심 주장을 구조화 | 주장(Claim) 객체와 탐지 모듈 |
| 공식 데이터 조회·매핑 | KOSIS 통계표 메타데이터를 임베딩·리랭커 기반 RAG로 검색 | 주장-통계표 자동 매핑 모듈 |
| 사실 여부 판별 | 시점·단위·모집단 정규화 후 수치 비교, 3분류 판정 | 판정 엔진(일치/불일치/판단불가) |
| 검증 결과 설명 생성 | 근거 통계·계산 과정·한계를 포함한 LLM 설명 생성 | 판정 리포트 및 설명 생성기 |
| 실서비스 고려 구조 | FastAPI 기반 API 서버 + 검증자 리뷰 흐름 | E2E 데모 서비스와 평가 리포트 |

> **핵심 원칙 ① 근거가 없으면 판정하지 않는다.**  
> 대응 통계가 없거나 정의가 불명확하면 **판단불가**로 회피합니다.

> **핵심 원칙 ② 모든 판정에는 출처가 따라붙는다.**  
> 어떤 통계표의 어떤 수치를 어떻게 계산했는지 기록하고 설명합니다.

# 1. 추진 배경 및 문제 정의

> **통계 왜곡은 반복적으로 발생하지만, 검증은 여전히 수작업입니다.**

뉴스와 온라인 콘텐츠에서는 수치를 다루는 과정에서 다양한 오류와 왜곡이 지속적으로 발생합니다. 그러나 현재의 검증은 담당자가 기사를 검색하고 공식 통계자료를 직접 확인하는 수작업 비중이 높아, 검증 누락과 대응 지연의 가능성이 상존합니다.

| 왜곡·오류 유형 | 설명 | 예시 |
|---|---|---|
| 수치 왜곡 | 공식 통계와 다른 수치를 인용 | 실업률 7.2%를 10%로 보도 |
| 시점 혼동 | 통계 기준 시점과 기사 서술 시점 불일치 | 2023년 통계를 “올해” 수치로 인용 |
| 단위 혼동 | 단위·척도를 잘못 해석 | 천 가구 단위를 가구 수로 표기 |
| 모집단 불일치 | 통계의 대상 범위와 기사 서술 범위가 다름 | 전체 농가 통계를 과수 농가로 서술 |
| 맥락 없는 발췌 | 일부 데이터만 선택적으로 인용해 해석 왜곡 | 증가 구간만 발췌해 급증으로 서술 |
| 과장 표현 | 수치 자체는 맞으나 표현이 통계적 의미를 초과 | 1% 감소를 “급감”으로 표현 |

> 필요한 것은 **“기사를 잘 요약하는 AI”가 아니라, 기사를 주장 단위로 분해하고 공식 통계와 대조하며, 판단 근거를 기록으로 남기는 업무형 검증 AI**입니다.

# 2. 프로젝트 목표 및 범위

> **실제 서비스 적용을 고려한 구조로, 검증의 전 과정을 하나의 흐름으로 연결합니다.**

## 2.1 목표

- 뉴스 기사 내 수치 기반 주장을 자동 탐지하고 유형(증감/규모/비교/전망)을 분류하는 모듈 구현
- KOSIS Open API 기반 공식 통계 조회·매핑 및 시점·단위·모집단 정렬 로직 구현
- 공식 통계 비교 기반 3분류 판정(일치/불일치/판단불가)과 불일치 유형 분석 구현
- 판정 근거와 한계를 포함한 LLM 설명 생성 및 검증자 리뷰 흐름 설계
- 골든셋 기반 정량·정성 평가 체계를 갖춘 End-to-End 서비스(API) 형태의 결과물 제출

## 2.2 범위 정의

| 구분 | 내용 |
|---|---|
| In-Scope (수행) | 수치 기반 주장 탐지, KOSIS 통계 조회·매핑, 비교 판정, 설명 생성, 검증자 리뷰 흐름, FastAPI 기반 데모, 골든셋 평가 |
| Out-of-Scope (제외) | 의견·가치판단 검증, 이미지·영상 팩트체크, KOSIS 외 비공식 출처 기반 검증, 실시간 대규모 크롤링, 완전 자동 판정(사람 리뷰 생략) |
| 데이터 범위 | 제공된 조선일보 기사 데이터셋(2025.01.01~2025.12.31), 검증 기준은 KOSIS 공식 통계로 한정 |

## 2.3 PoC 성공 기준

아래 수치는 골든셋 구축 후 난이도에 맞춰 멘토와 협의하여 확정하는 초기 목표치입니다.

| 영역 | 성공 기준(초기 목표) |
|---|---|
| 주장 탐지 | 골든셋 기준 검증 가능 주장 탐지 F1 ≥ 0.80 |
| 통계 매핑 | 주장-통계표 매핑 Hit@3 ≥ 0.70 (상위 3개 후보 내 정답 포함) |
| 판정 정확도 | 3분류 판정 정확도 ≥ 0.75, 판단불가(Fallback) 정확도 ≥ 0.80 |
| 설명 품질 | 설명 내 근거 수치 인용 정확률 ≥ 0.90 (환각 수치 0건 목표) |
| 서비스 완결성 | 기사 URL/본문 입력 → 판정 리포트 출력까지 E2E 동작하는 API/데모 |

# 4. 검증 파이프라인 설계

> **탐지 → 분류 → 추출 → 조회 → 판정 → 설명 → 리뷰의 9단계 흐름**

| 단계 | 처리 내용 | 출력 |
|---|---|---|
| ① 기사 입력·전처리 | 기사 본문 정제, 문장 분리, 메타데이터(작성일·섹션) 정리 | 정제된 문장 리스트 |
| ② 주장 탐지 | 수치·비교 표현 포함 문장 필터링(규칙) 후 LLM으로 검증 가능 주장 선별 | 주장 후보 문장 |
| ③ 주장 유형 분류 | 증감 / 규모 / 비교 / 전망 4개 유형으로 분류(전망형은 검증 대상 제외 처리) | 유형 라벨 |
| ④ 핵심 정보 추출 | 지표명, 시점(기간), 단위, 모집단, 수치값, 비교 대상을 구조화(JSON) | Claim 객체 |
| ⑤ 공식 통계 조회 | KOSIS 통계목록·메타정보 검색 → 후보 통계표 선정 → 통계자료 API로 수치 조회 | 근거 통계 데이터 |
| ⑥ 존재 여부 분기 | 대응 통계 부재 또는 정의 불명확 시 “판단불가”로 분기 | 검증 가능/불가 분기 |
| ⑦ 비교 판정 | 시점·단위·모집단 정렬 후 수치 비교(파생 계산 포함), 허용 오차 내 일치 여부 판단 | 일치/불일치 판정 |
| ⑧ 설명 생성 | 판정 근거, 사용 통계표, 계산 과정, 판정의 한계를 포함한 설명 생성(LLM) | 판정 리포트 |
| ⑨ 검증자 리뷰 | 사람이 판정·설명을 확인하고 보정, 최종 확정. 보정 이력은 개선 데이터로 축적 | 최종 판정 결과 |

**핵심 데모 흐름**  
기사 입력 → 주장 카드 → 근거 통계·계산 과정 → 판정 배지(일치/불일치/판단불가) → 검증자 승인 → 결과 리포트

# 5. 판정 체계

> **판정은 3분류로 명확히 하고, 불일치는 유형까지 분석합니다.**

## 5.1 판정 3분류

| 판정 | 기준 | 예시 |
|---|---|---|
| 일치 | 기사 수치가 공식 통계와 동일하거나, 시점·단위 기준으로 정렬했을 때 허용 오차 내 일치 | 기사 “실업률 7.2%” ↔ 공식 데이터 7.2% |
| 불일치 | 기사 수치가 공식 데이터와 다르거나, 시점·단위·모집단 해석이 잘못된 경우 | 기사 “실업률 10%” ↔ 공식 데이터 7.2% |
| 판단불가 | 지표·시점이 불명확하거나 대응되는 공식 통계가 없는 경우(억지 판정 금지) | “경제 상황이 크게 악화되었다” |

## 5.2 불일치 유형 분석

| 불일치 유형 | 판정 사유 예시 |
|---|---|
| 수치 불일치 | 동일 지표·시점·모집단 기준에서 수치 자체가 허용 오차를 초과 |
| 기간 불일치 | 기사가 인용한 시점과 통계 수록 시점이 다름 |
| 모집단 불일치 | 통계의 조사 대상과 기사 서술 대상이 다름 |
| 과장 표현 | 수치는 근사하나 표현 강도가 통계적 변화를 과장 |

## 5.3 판정 규칙

- **허용 오차:** 반올림·유효자릿수를 고려한 상대 오차 기준을 사전에 정의하고 골든셋으로 캘리브레이션
- **단위 환산:** %, %p, 천 단위, 면적(ha·㎢) 등 단위 정규화 후 비교
- **파생 계산 재현:** 기사 수치가 원자료의 가공값(예: 고령화 비율 = 65세 이상 합계 ÷ 전체)일 경우 계산 과정을 재현하여 검증
- **Fallback 우선:** 근거가 부족하면 불일치가 아니라 판단불가로 분류 - 잘못된 확신보다 회피가 낫다는 원칙

### 예시: 클라비 제시 사례

“농가 고령화에 올 과일 재배면적 1% 줄어” 기사의 고령화 주장 검증 시, KOSIS 농림어업조사 통계표에서 2024년 과수 농가 **166,558가구** 중 65세 이상 4개 구간 합계 **106,877가구**를 재현 계산하여:

`106,877 ÷ 166,558 ≈ 64.2%`

로 기사 수치와의 일치를 판정합니다. 이처럼 단순 수치 대조가 아닌 **계산 재현형 검증**까지 포함합니다.

# 6. 기술 아키텍처

> **제공된 HCX 모델군을 작업 특성에 맞게 역할 분담하여 비용·품질을 균형 있게 설계합니다.**

## 6.1 모듈 구성

| 모듈 | 역할 | 핵심 기술 |
|---|---|---|
| Ingest | 기사 데이터 로딩·정제·문장 분리 | Python, pandas |
| Claim Detector | 수치 문장 필터링, 검증 가능 주장 탐지·유형 분류 | 규칙 기반 필터 + HCX |
| Claim Parser | 지표·시점·단위·모집단·수치 구조화 추출 | HCX(JSON 출력) |
| KOSIS Retriever | 통계표 검색·매핑, 수치 조회·가공 | KOSIS Open API, 임베딩, 벡터 DB, 리랭커 |
| Verdict Engine | 정규화·비교·파생 계산·3분류 판정·불일치 유형 분석 | Python 수치 로직 + RAG Reasoning 보조 |
| Explainer | 판정 근거·계산 과정·한계 설명 생성 | HCX-003/005/007 |
| Review & Serve | API 서버, 검증자 리뷰 화면, 결과 저장 | FastAPI, 웹 데모 |

## 6.2 제공 리소스(HCX) 활용 계획

| 제공 모델 | 활용 용도 |

# 8. 평가 계획 - 골든셋 기반 신뢰성 검증

> **모델 점수가 아니라, 검증 업무가 실제로 신뢰 가능한지를 평가합니다.**

## 8.1 골든셋 설계

| 항목 | 계획 |
|---|---|
| 규모(목표) | 기사 약 100건, 검증 가능 주장 약 200~300개(구축 여건에 따라 조정) |
| 라벨 | 주장 여부, 주장 유형, 정답 통계표 ID, 판정(일치/불일치/판단불가), 불일치 유형 |
| 구축 방법 | 팀원 교차 라벨링 + 불일치 항목 합의 프로세스, 라벨링 가이드 문서화 |
| 분포 | 일치·불일치·판단불가가 모두 포함되도록 균형 구성(판단불가 케이스 의도적 포함) |

## 8.2 정량 지표

| 단계 | 지표 |
|---|---|
| 주장 탐지 | Precision / Recall / F1 |
| 유형 분류 | 유형별 Accuracy, Macro-F1 |
| 통계 매핑 | Hit@1 / Hit@3(정답 통계표 포함률), 매핑 실패율 |
| 판정 | 3분류 Accuracy, 클래스별 F1, 판단불가(Fallback) 정확도 |
| 설명 | 근거 수치 인용 정확률, 환각(근거 없는 수치) 발생률, 설명 내 출처 제시율 |
| 서비스 | E2E 처리 시간, API 성공률, 검증자 보정률(사람이 뒤집은 판정 비율) |

## 8.3 정성 평가와 개선 루프

- **설명 품질 루브릭:** 근거 명시성, 계산 과정 재현성, 한계 고지 여부를 3점 척도로 평가
- **오류 분석:** 실패 사례를 탐지 실패/매핑 실패/정렬 실패/판정 오류/설명 오류로 분류해 개선 우선순위 도출

> **평가 → 실패 유형 분석 → 프롬프트·검색·판정 규칙 개선 → 재평가**가 반복되는 운영 루프를 프로젝트 기간 내내 유지합니다.

|---|---|
| 데이터 확장 | KOSIS 외 공공 Open API(한국은행 ECOS, 공공데이터포털 등)로 검증 근거 확대 |
| 업무 확장 | 기사 작성 전 데스크 검수 도구, 보도자료·정책자료 검증, 기업 홍보물 수치 검수 |
| 서비스 확장 | 언론사·팩트체크 기관용 SaaS, 공공기관 보도 모니터링 대시보드 |
| 클라비 자산 연계 | ClaRIO(RAG)·ClaRIO-EVA(평가) 구조와 결합해 Trust 기반 Vertical 서비스로 발전 |
