## 1. 문제 정의

ClaFact는 뉴스 기사 속 수치 주장을 KOSIS(국가통계) 공식 통계로 자동 검증하고, 근거·계산과정·한계까지 설명하는 AI 기반 사실검증 시스템이다. 본 프로젝트가 풀고자 하는 문제는 뉴스와 콘텐츠에서 수치 주장이 빠르게 유통되는 반면, 그 수치가 공식 통계와 일치하는지, 같은 시점·단위·모집단을 기준으로 말하고 있는지 검증하는 과정은 여전히 수작업에 의존한다는 점이다.

뉴스의 수치 주장은 독자에게 강한 신뢰감을 준다. 그러나 수치 자체가 틀린 경우뿐 아니라, 통계 기준 시점이 다르거나, 모집단 정의가 다르거나, 단위가 다르게 해석되는 경우에도 사실과 다른 인상을 줄 수 있다. 예를 들어 “전체 농가” 통계와 “과수 농가” 통계를 혼용하거나, 특정 연도 자료를 현재 상황처럼 표현하면 수치가 존재하더라도 주장은 검증 가능한 방식으로는 불완전해진다.

따라서 ClaFact의 핵심 가치는 단순히 기사를 요약하거나 관련 문서를 검색하는 데 있지 않다. ClaFact는 기사를 Claim 단위로 구조화하고, KOSIS 공식 통계를 Evidence로 연결한 뒤, 결정적 코드 기반 Verdict를 통해 어디까지 판단할 수 있는지 명시한다. 이때 판정 결과뿐 아니라 사용한 통계표, 수치, 계산식, 판정 한계까지 함께 제시함으로써 재현성과 감사가능성을 확보한다.

본 프로젝트에서 가장 큰 리스크는 모델 성능 자체가 아니라 “기사 속 주장과 공식 통계의 정의 불일치”이다. 같은 단어처럼 보이는 지표도 KOSIS 통계표에서는 조사 대상, 분류 경로, 수록 기간, 단위, 모집단이 다를 수 있다. 따라서 ClaFact는 근거가 부족하면 억지 판정을 하지 않고 “판단불가”로 회피하는 것을 정상 결과로 취급한다. 이는 실패가 아니라, Evidence보다 강한 Verdict를 내리지 않기 위한 신뢰성 원칙이다.

본 아이펠톤 기간(2026.07.22~09.09, 약 7주) 동안의 목표는 전체 사실검증 플랫폼을 완성하는 것이 아니라, “KOSIS 공식 통계 기반 뉴스 수치 Claim 검증”이라는 1차 Vertical PoC를 실제로 작동하게 만드는 것이다. 제안서 제출은 2026.07.20, PoC 발표는 2026.07.21로 예정되어 있으며, 본 PoC는 클라비(CLABI) 내부 연구과제의 선행 검증 성격을 가진다.

## 2. 배경지식 및 참고자료

자동 팩트체크(automated fact-checking)는 일반적으로 Claim을 식별하고, 관련 Evidence를 찾고, Evidence에 근거해 Claim의 진위를 판단하는 흐름으로 구성된다. ClaFact 역시 Content → Claim → Evidence → Verdict → Service → Evaluation → Improvement의 구조를 따른다. 다만 본 프로젝트는 범위를 뉴스 기사 속 수치 주장과 KOSIS 공식 통계로 제한함으로써, 추상적인 일반 지식 검증이 아니라 재현 가능한 통계 기반 검증을 목표로 한다.

Claim detection/extraction은 기사 본문에서 검증 가능한 문장을 찾고, 그 문장을 구조화된 Claim 객체로 바꾸는 과정이다. ClaFact에서는 수치·단위·시점·비교 표현을 포함한 문장을 우선 탐지하고, Claim의 지표명, 수치값, 단위, 기준 시점, 모집단, 비교 대상을 추출한다. 현재 코드베이스에는 규칙 탐지(detect), 수치·단위·시점 파싱(parse), 상태머신 기반 schemas가 이미 구현되어 있다.

Evidence retrieval은 Claim과 대응되는 근거 데이터를 찾는 단계이다. 일반적인 RAG에서는 문서 검색과 임베딩 검색이 중요한 역할을 하지만, ClaFact의 병목은 단순 문서 검색보다 KOSIS 통계표 매핑에 있다. KOSIS `statisticsList.do`는 일반 검색 API라기보다 `parentListId` 기반 트리 탐색 구조이므로, 통계표 4,602개를 대상으로 사전 인덱스를 구축해야 한다. 또한 `TBL_NM`(통계표명)만으로는 충분하지 않으며, 분류경로(breadcrumb), 메타정보, 수록기간, 단위, 모집단 정보를 함께 인덱싱해야 Hit@3 개선을 기대할 수 있다.

Numerical claim verification은 Claim의 수치와 Evidence의 수치를 비교하는 단계이다. 이때 단순 문자열 대조가 아니라 단위 환산, 반올림 허용오차, 시점 정렬, 모집단 일치 여부, 파생계산 재현이 필요하다. ClaFact의 대표 사례는 과수농가 166,558가구 중 65세 이상 106,877가구라는 KOSIS 근거를 사용해 `106,877 ÷ 166,558 = 64.2%`를 재현 계산하고, 기사 주장과 일치한다고 판단하는 경우이다.

참고할 만한 연구 흐름으로는 FEVER와 같은 fact verification 데이터셋·태스크, 표(table) 또는 통계 데이터 기반 검증, RAG 기반 근거 검색, Claim extraction 및 evidence retrieval 평가가 있다. 다만 본 제안서에서는 구체 논문 제목·저자·연도를 확신 없이 기재하지 않는다. 실제 최종 제안서에는 관련 연구의 정확한 제목, 연도, 태스크 정의, ClaFact와의 차별점을 별도 문헌조사 후 보강해야 한다. [확인 필요: 논문 서치]

## 3. 사용할 데이터

본 프로젝트의 기사 데이터는 조선일보 2025.01.01~2025.12.31 기사 데이터셋이다. 제공 예정 데이터의 필드는 제목, 작성일, 섹션, URL, 본문, 검색구분 레이블이다. 검색구분 레이블은 “국가데이터처/통계청” 키워드 포함 여부에 따라 True/False로 제공되는 것으로 정의되어 있다. 단, 현재 시점에는 조선일보 기사 데이터셋이 아직 도착하지 않았으므로, 데이터 도착 전에는 Fixture 기반 예제 기사와 골든셋 시드로 개발을 진행한다.

근거 데이터는 KOSIS Open API 4종과 오프라인 통계표 목록 xls를 사용한다. KOSIS Open API는 통계목록, 통계자료, 메타정보, 통계설명으로 구성된다. 통계목록 API는 통계표 후보 탐색과 인덱스 구축의 출발점이며, 통계자료 API는 실제 수치 조회에 사용된다. 메타정보와 통계설명은 지표 정의, 분류, 단위, 모집단, 수록기간을 확인해 Claim과 Evidence의 정의 불일치를 줄이는 데 필요하다.

현재 HCX API Key, KOSIS API Key, 조선일보 기사 데이터셋은 아직 미도착 상태이다. 따라서 개발 순서는 API Key가 없어도 가능한 작업을 먼저 수행한다. 오프라인 통계표 목록 xls 4,602개를 이용해 통계표 인덱스를 선행 구축하고, 골든셋 Claim에 대해 정답 통계표 ID를 라벨링한 뒤, Hit@k 평가를 먼저 시작할 수 있다. KOSIS API Key가 도착하면 통계자료·메타정보·통계설명 조회를 실제 API로 전환한다.

골든셋은 현재 12건 시드가 존재하며, 1차 목표는 50건으로 확장하는 것이다. 현재 12건 기준으로 탐지 F1은 0.9412에서 실패 자동기록, 규칙 A2-0003 작성 및 테스트를 거쳐 1.0000으로 개선되었다. 그러나 이 성능은 12건 시드 기준이라는 한계가 있으므로, 제안서와 발표에서 과장하지 않는다. 50건 확장 시에는 라벨러와 규칙작성자를 분리하고, 교차 라벨링 및 불일치 합의 절차를 둔다.

데이터 취득 계획은 다음과 같다.

| 구분 | 예정 데이터 | 도착 전 대체 | 도착 후 작업 |
|---|---|---|---|
| 기사 데이터 | 조선일보 2025년 기사 데이터셋 | Fixture 기사, 12건 골든셋 시드 | 기사 EDA, Claim 후보 추출, 골든셋 50건 확장 |
| KOSIS 근거 | Open API 4종 | 오프라인 통계표 목록 xls 4,602개 | 통계자료·메타정보·통계설명 조회 연동 |
| HCX 모델 | DASH, HCX, Reranker, Embedding v1/v2 | Fixture↔Http 스위치, 규칙 기반 처리 | Claim 추출, 설명 생성, 임베딩·리랭커 검색 실험 |

## 4. 문제 해결 방법

ClaFact의 문제 해결 방법은 9단계 파이프라인으로 구성된다.

| 단계 | 내용 | 주요 출력 |
|---|---|---|
| 1. 입력 | 기사 본문 또는 URL 입력 | 원문 Content |
| 2. 전처리 | 본문 정제, 문장 분리, 메타데이터 정리 | 문장 리스트 |
| 3. 주장탐지(규칙+LLM) | 수치·비교 표현 기반 후보 탐지 후 LLM 보조 선별 | Claim 후보 |
| 4. 유형분류 | 규모, 비율, 증감, 비교, 파생계산 등 유형 분류 | Claim type |
| 5. 핵심정보추출 | 지표명, 시점, 단위, 모집단, 수치, 비교 대상 추출 | 구조화 Claim |
| 6. KOSIS조회 | 통계표 후보 검색, 메타 확인, 수치 조회 | Evidence 후보 |
| 7. 분기 | 근거 부재·정의 불명확 시 판단불가 처리 | 검증 가능/불가 판단 |
| 8. 비교판정(결정적) | Python 로직으로 정규화·계산·허용오차 비교 | 일치/불일치/판단불가 |
| 9. 설명생성→검증자리뷰 | 근거·계산·한계 설명 생성 후 사람 검토 | 판정 리포트 |

중요한 제약은 최종 판정을 LLM에게 맡기지 않는다는 점이다. LLM은 탐지, 추출, 설명, 검색 보조에 사용할 수 있지만, 최종 일치/불일치/판단불가는 Python 결정적 로직이 수행한다. 이 구조는 재현성, 감사가능성, 회귀 테스트 가능성을 확보하기 위한 것이다.

기존 코드베이스는 최대한 재사용한다. 현재 완료된 모듈은 schemas(상태머신), detect(규칙 탐지), parse(수치·단위·시점 파싱), retrieve(경로A 별칭+키워드), verdict(결정적 판정 엔진), kosis/llm(Fixture↔Http 스위치), eval 하네스와 metrics, 골든셋 12건 시드, 규칙카드, 실패로그, Dockerfile이다. 따라서 새로 집중해야 할 빈 영역은 KOSIS 인덱싱과 검색이다.

KOSIS 검색은 두 경로로 비교한다. 경로A는 별칭사전과 키워드 기반 검색이다. 이는 구현이 빠르고 실패 원인을 사람이 해석하기 쉽다는 장점이 있다. 경로B는 Embedding v1/v2와 Reranker를 사용하는 검색이다. 이는 표현이 다른 Claim과 통계표를 연결할 가능성이 있지만, 인덱스 품질과 라벨링된 평가셋이 필요하다. 두 경로는 EXP-001로 정의하여 동일 골든셋에서 Hit@1, Hit@3, 실패 유형을 비교한다.

인덱스 구축 순서는 다음과 같다. 첫째, 오프라인 xls 4,602개 통계표 목록을 읽어 통계표 ID, 통계표명, 기관, 분류경로를 표준 스키마로 정리한다. 둘째, 가능한 경우 breadcrumb와 메타정보(수록기간, 단위, 모집단)를 결합한 검색 문서를 만든다. 셋째, 골든셋 Claim마다 정답 통계표 ID를 라벨링한다. 넷째, Hit@k를 측정하는 평가 하네스를 만든다. 다섯째, 경로A와 경로B를 같은 평가셋에서 비교하고 실패를 별칭사전, 규칙카드, 골든셋 보강으로 전환한다.

현 시점에서는 API Key와 실제 기사 데이터셋이 미도착이므로 정밀한 검색 방법을 확정하기 어렵다. 그러나 일반적 접근은 명확하다. 먼저 오프라인 인덱스와 결정적 판정 흐름을 완성하고, 이후 KOSIS API와 HCX 모델이 도착하면 Fixture 모드에서 Http 모드로 전환한다. 이 방식은 외부 리소스 지연에도 개발을 멈추지 않고, 실제 연동 시 회귀 테스트로 안정성을 확인할 수 있다.

## 5. 평가 방법

ClaFact의 평가는 단순 정확도 하나로 판단하지 않는다. Claim 탐지, KOSIS 매핑, Verdict 판정, 판단불가 회피, 설명 인용 정확성을 단계별로 분리해 측정한다. 이는 Claim, Evidence, Verdict를 섞지 않는 프로젝트 원칙과도 연결된다.

양적 평가는 골든셋 기반 자동 평가 하네스로 수행한다. 초기 목표치는 다음과 같다. 단, 이 목표치는 골든셋 50건 이상 구축 후 멘토와 협의해 확정할 초기값이며, 현재 성능을 보장하는 수치가 아니다.

| 평가 영역 | 지표 | 초기 목표 |
|---|---|---|
| 주장탐지 | Precision / Recall / F1 | F1 ≥ 0.80 |
| 통계매핑 | Hit@1 / Hit@3 | Hit@3 ≥ 0.70 |
| 판정 | 3분류 판정정확도 | Accuracy ≥ 0.75 |
| 판단불가 | Fallback 정확도 | Accuracy ≥ 0.80 |
| 설명 | 근거 수치 인용정확률 | ≥ 0.90, 환각 0건 목표 |

현재 탐지 F1 1.0000은 12건 시드 기준이다. 따라서 이를 전체 성능으로 주장하지 않는다. 제안서와 발표에서는 “12건 기준 1회 플라이휠에서 실패 기록 → 규칙 작성 → 테스트 → F1 개선이 실제로 작동했다”는 프로세스 검증 결과로만 제시한다. 골든셋이 50건으로 확장되면 전회차 diff를 통해 어떤 규칙이 어떤 성능 변화를 만들었는지 추적한다.

질적 평가는 판정 리포트 예시와 실패 전환 흐름을 중심으로 한다. 리포트에는 최소한 Claim 원문, 구조화된 Claim, 사용한 KOSIS 통계표, Evidence 수치, 계산 과정, 판정 결과, 판정 한계, 판단불가 사유가 포함되어야 한다. 특히 “근거가 없으면 판단불가로 회피한다”는 원칙이 실제로 작동하는 사례를 포함한다.

대표 판정 예시는 과수농가 고령화 비율 검증이다. Evidence로 과수농가 전체 166,558가구와 65세 이상 106,877가구가 확보되면, Verdict 엔진은 `106,877 ÷ 166,558 = 64.2%` 계산을 재현한다. 이 값이 기사 주장과 허용오차 내에서 일치하면 “일치”로 판정한다. 단, 이 과정에서도 통계표의 기준 연도, 모집단, 단위가 Claim과 맞는지 확인해야 하며, 정의가 불명확하면 “판단불가”로 회피한다.

비교 평가는 경로A(별칭+키워드)와 경로B(임베딩+리랭커)의 Hit@k 비교로 수행한다. 단순히 어떤 방법의 평균 점수가 높은지만 보지 않고, 실패 사례를 매핑 실패, 메타 부족, 모집단 불일치, 시점 불일치, Claim 추출 오류로 분류한다. 각 실패 1건은 별칭사전 1줄, 규칙카드 1개, 골든셋 보강 1건, 또는 판단불가 규칙으로 전환한다.
